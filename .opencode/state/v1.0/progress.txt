## Codebase Patterns
<!-- Consolidate reusable patterns here -->
- Public API is curated explicitly through `src/index.ts`, `src/testkit.ts`, and `src/runtime.ts` with named re-exports only
- Internal seam types can stay exported from internal barrels for tests and implementation, but the public subpath barrels should only re-export stable consumer-facing types and factories
- Runtime compatibility work lives in `src/runtime/` and `src/runtime/detect.ts`; core modules stay runtime-agnostic behind `RuntimeAdapter`
- Cross-cutting CLI pipeline changes usually thread through `src/core/cli/index.ts`, `src/core/testkit/index.ts`, and `src/core/resolve/index.ts`
- CLI-wide lifecycle extensions live in `src/core/cli/plugin.ts` and are threaded through `RunOptions.plugins`; hooks wrap parse, resolve, and action stages outside command middleware in registration order
- Runtime support policy should live in one internal source (`src/runtime/support.ts`) and feed adapter guards, docs assertions, and `package.json` engine checks to avoid version drift
