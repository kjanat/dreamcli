## Codebase Patterns
<!-- Consolidate reusable patterns here -->

- **Immutable builders**: every modifier returns `new XBuilder({...this.schema, changed})`
- **Phantom types**: `declare readonly _config: C` for type-level tracking, no runtime cost
- **Schema/builder split**: `XSchema` for runtime, phantom generics for compile-time
- **Coercion pattern**: discriminated `{ ok: true, value } | { ok: false, error }` results
- **Type erasure**: `ErasedCommand` for heterogeneous storage, typed closures preserve safety
- **Error architecture**: `CLIError` base → `ParseError`, `ValidationError` subtypes with stable codes
- **Test co-location**: tests in same directory as source, split across files for large modules
- **Factory pattern**: `flag.string()`, `arg.string()`, `command(name)`, `cli(name)`
- **Zero runtime deps**: all devDependencies only
- **Imports**: relative paths with `.js` extension, not barrel imports in tests

- **WidenContext conditional type**: `Record<string, never>` as default C makes naive intersection collapse to `never`. Use `WidenContext<C, Output> = C extends Record<string, never> ? Output : C & Output` to replace on first middleware, intersect on subsequent.
- **Middleware explicit generic**: `middleware<{ user: string }>(...)` is needed for reliable inference in strict TS mode; inference from `next()` argument alone is unreliable across all TS contexts.
- **Middleware phantom type via cast**: `MiddlewareImpl` (runtime) cast to `Middleware<Output>` (phantom-branded) — same pattern as flag/arg builders.
- **executeWithMiddleware continuation chain**: Build from back to front so first middleware is outermost. Each `next()` closes over downstream chain.
- **Global flag pattern**: Detect in `CLIBuilder.execute()`, strip from argv before dispatch, propagate via options to downstream `runCommand()`. No global flag schema — hardcoded like `--version`/`--help`.
- **JSON mode redirection**: In JSON mode, `log`/`info` → stderr, `json()` → stdout. Errors rendered via `out.json({ error: err.toJSON() })` to stdout. Scripts get one structured JSON stream on stdout.
- **Table output dual-mode**: `table()` in JSON mode delegates to `json(rows)` (full data, unfiltered). In normal mode, delegates to `log(formatTable(...))` so redirection rules apply automatically. Column ordering/filtering is display-only.
- **isTTY wiring pattern**: `adapter.isTTY` → `CLIRunOptions.isTTY` → `RunOptions.isTTY` → `createCaptureOutput({ isTTY })` → `OutputChannel.isTTY` → `Out.isTTY`. Each layer conditionally spreads to respect `exactOptionalPropertyTypes`. `.run()` auto-sources from adapter when not explicitly set.
- **Output mode precedence**: `jsonMode > isTTY` — handlers check `jsonMode` first, then `isTTY`. The `isTTY` field preserves raw TTY status even in JSON mode (handlers may need it for edge cases), but jsonMode always takes precedence for decorative output decisions.
- **CLIBuilder.command() must accept any C**: `eraseCommand()` and `CLIBuilder.command()` need `C extends Record<string, unknown>` generic param so commands with middleware (which widen C beyond `Record<string, never>`) can be registered.
- **JSON error rendering at every error site**: Both `CLIBuilder.execute()` (dispatch errors) and `runCommand()` (command errors) have separate error paths. Each must check `jsonMode` and use `out.json({ error: err.toJSON() })` vs `out.error(err.message)`. Easy to miss dispatch-level errors since they're not in the try/catch of runCommand.
