# Progress Log
PRD: v0.7-subcommand-nesting
Started: 2026-02-10

## Codebase Patterns
- **Adding FlagSchema fields**: add to `FlagSchema` interface, add default in `createSchema()`,
  add method on `FlagBuilder` returning `new FlagBuilder({...this.schema, field: value})`.
  Metadata-only fields (no type-level change) return `FlagBuilder<C>` unchanged.
- **Manual FlagSchema construction** exists in `completion.test.ts` `flagSchema()` helper тАФ must
  update when adding new required fields.
- **Manual CommandSchema construction**: 8 test files have `makeSchema()`/`commandSchema()` helpers.
  New required fields on `CommandSchema` must be added to all of them: `parse.test.ts`,
  `resolve.test.ts`, `resolve-prompt.test.ts`, `resolve-env.test.ts`, `resolve-config.test.ts`,
  `resolve-interactive.test.ts`, `resolve-errors.test.ts`, `completion.test.ts`.
- **Test pattern**: mirror `.deprecated()` test structure тАФ runtime checks + `expectTypeOf` for
  type preservation across all presence states.
- **Adding CommandBuilder methods**: all `new CommandBuilder(...)` calls in class methods must pass
  `this._subcommands` as third arg to preserve nested commands across chaining. Methods that
  don't change F/A/C preserve handler; methods that do (flag, arg, middleware) drop handler but
  keep subcommands.
- **Propagated flag collection**: `collectPropagatedFlags(commandPath)` in `cli/propagate.ts` тАФ
  takes `CommandSchema[]` (rootтЖТtarget), returns inherited `Record<string, FlagSchema>`. Target's
  own flags always shadow. Dispatch layer merges result into target's flags before resolution.
- **Recursive dispatch**: `dispatch()` in `cli/dispatch.ts` тАФ walks argv consuming command name
  tokens, returns discriminated union (`match` | `needs-subcommand` | `unknown`). Commands with
  both handlers + subcommands: if next token matches subcommand тЖТ descend; else тЖТ match at current
  level. `findClosestCommand()` + `levenshtein()` also live in dispatch.ts.
- **Completion tree walking**: `walkCommandTree(schemas, ancestorPath)` in `completion/index.ts` produces
  flat `CommandNode[]` with `mergedFlags` (propagated + own). Both bash/zsh generators consume this. Bash
  uses path-keyed case statements; zsh uses per-group helper functions.
- **ErasedCommand.subcommands**: `ReadonlyMap<string, ErasedCommand>` keyed by name AND alias.
  `eraseCommand()` builds recursively from `_subcommands`. Test helpers need `subcommands: new Map()`.
- **mergedSchema on RunOptions**: `@internal` field тАФ when dispatch merges propagated flags, it
  passes the merged schema via `RunOptions.mergedSchema`. `runCommand()` uses this for parse/resolve
  instead of `cmd.schema` when present.

## Task - propagated-flag-collection
- Created `src/core/cli/propagate.ts` with `collectPropagatedFlags()` pure function
- Input: `commandPath: readonly CommandSchema[]` (root ancestor тЖТ target, inclusive)
- Algorithm: walk ancestors root-first, collect `propagate: true` flags; later ancestors overwrite earlier (natural shadowing); target's own flags (any) delete from result
- Returns only *inherited* flags тАФ target's own flags excluded (caller merges)
- 16 tests across 6 describe groups: base cases, single ancestor, multi-level, target shadowing, intermediate shadowing, return value properties
- **Files changed:** `src/core/cli/propagate.ts` (new), `src/core/cli/cli-propagate.test.ts` (new)
- **Learnings:**
  - Intermediate non-propagated redefinitions do NOT block ancestor propagation to grandchildren тАФ only `propagate: true` flags flow through the accumulator. Intermediate shadowing only applies when the intermediate IS the target.
  - `delete` on accumulated record is fine for sparse cleanup тАФ biome doesn't flag `noDelete` here (only on prototype properties)
  - Function is intentionally NOT exported from CLI barrel or public API тАФ it's @internal infrastructure for the dispatch layer (`recursive-dispatch` task will import directly)

- Handles hybrid commands (action + subcommands): subcommand match takes priority, else falls to group handler
- Groups without handlers: `needs-subcommand` тЖТ renders group help with subcommand list
- Updated `completion.test.ts` `erased()` helper with `subcommands: new Map()`
- 22 unit tests in `cli-dispatch.test.ts`: base cases, nesting, groups, levenshtein, findClosest, uniqueCommands
- 16 integration tests in `cli-nesting.test.ts`: nested dispatch, 3-level, hybrid groups, propagated flags, help, JSON mode, aliases

- **Learnings:**
  - Flags before command names (e.g. `--verbose db migrate`) require scanning past flags to find the first non-flag token as the command name. The dispatch function handles this by finding the first non-`-` prefixed token.
  - Hybrid commands (with action + subcommands, like `git remote`) need the dispatch to try subcommand matching first, then fall back to the parent handler for unknown tokens that could be positional args.
  - `ReadonlyMap` works well for the subcommand lookup table тАФ O(1) per level, alias entries just point to the same object. `uniqueCommands()` deduplicates for suggestion/help rendering.
  - `mergedSchema` on RunOptions is a clean internal seam тАФ doesn't change the public API, and `runCommand()` transparently uses it for parse/resolve while keeping middleware from `cmd.schema`.
