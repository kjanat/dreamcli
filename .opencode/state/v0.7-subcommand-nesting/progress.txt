# Progress Log
PRD: v0.7-subcommand-nesting

## Codebase Patterns
- **Adding FlagSchema fields**: add to `FlagSchema` interface, add default in `createSchema()`,
  add method on `FlagBuilder` returning `new FlagBuilder({...this.schema, field: value})`.
  Metadata-only fields (no type-level change) return `FlagBuilder<C>` unchanged.
- **Manual FlagSchema construction** exists in `completion.test.ts` `flagSchema()` helper — must
  update when adding new required fields.
- **Manual CommandSchema construction**: 8 test files have `makeSchema()`/`commandSchema()` helpers.
  New required fields on `CommandSchema` must be added to all of them: `parse.test.ts`,
  `resolve.test.ts`, `resolve-prompt.test.ts`, `resolve-env.test.ts`, `resolve-config.test.ts`,
  `resolve-interactive.test.ts`, `resolve-errors.test.ts`, `completion.test.ts`.
- **Test pattern**: mirror `.deprecated()` test structure — runtime checks + `expectTypeOf` for
  type preservation across all presence states.
- **Adding CommandBuilder methods**: all `new CommandBuilder(...)` calls in class methods must pass
  `this._subcommands` as third arg to preserve nested commands across chaining. Methods that
  don't change F/A/C preserve handler; methods that do (flag, arg, middleware) drop handler but
  keep subcommands.
- **Propagated flag collection**: `collectPropagatedFlags(commandPath)` in `cli/propagate.ts` —
  takes `CommandSchema[]` (root→target), returns inherited `Record<string, FlagSchema>`. Target's
  own flags always shadow. Dispatch layer merges result into target's flags before resolution.

## Task - propagated-flag-collection
- Created `src/core/cli/propagate.ts` with `collectPropagatedFlags()` pure function
- Input: `commandPath: readonly CommandSchema[]` (root ancestor → target, inclusive)
- Algorithm: walk ancestors root-first, collect `propagate: true` flags; later ancestors overwrite earlier (natural shadowing); target's own flags (any) delete from result
- Returns only *inherited* flags — target's own flags excluded (caller merges)
- 16 tests across 6 describe groups: base cases, single ancestor, multi-level, target shadowing, intermediate shadowing, return value properties
- **Files changed:** `src/core/cli/propagate.ts` (new), `src/core/cli/cli-propagate.test.ts` (new)
- **Learnings:**
  - Intermediate non-propagated redefinitions do NOT block ancestor propagation to grandchildren — only `propagate: true` flags flow through the accumulator. Intermediate shadowing only applies when the intermediate IS the target.
  - `delete` on accumulated record is fine for sparse cleanup — biome doesn't flag `noDelete` here (only on prototype properties)
  - Function is intentionally NOT exported from CLI barrel or public API — it's @internal infrastructure for the dispatch layer (`recursive-dispatch` task will import directly)
