# Progress Log
PRD: v0.6-config-custom-deprecated
Started: 2026-02-10

## Codebase Patterns

- **RuntimeAdapter extension pattern**: Add field to interface → add to TestAdapterOptions with sensible default → implement in createTestAdapter → implement in createNodeAdapter → add `platform` to NodeProcess if needed → update all mock NodeProcess objects in tests → add test section for new fields
- **Homedir/configDir from env vars**: Avoid `node:os` import; derive from `process.env` (HOME, USERPROFILE, XDG_CONFIG_HOME, APPDATA) + `process.platform` to keep factory synchronous
- **readFile ENOENT → null**: Use `isNodeSystemError` type guard to detect `{ code: 'ENOENT' }` without `@types/node`; rethrow all other errors
- **node-builtins.d.ts**: Add minimal type shims for any new `node:*` imports (only the subset actually used)
- **Config discovery pattern**: Pure function taking `Pick<RuntimeAdapter, 'readFile' | 'cwd' | 'configDir'>` — narrow adapter pick for testability. Discriminated union result (`found: true/false`), errors thrown as `CLIError` with specific codes. Extension-based loader dispatch with JSON built-in.
- **Path separator inference**: Detect `\` vs `/` from adapter-provided paths (already platform-native) to avoid `node:path` import. `joinPath(base, ...segments)` helper.
- **Virtual filesystem testing**: `stubAdapter({ '/path': 'content' })` pattern for config tests — maps paths to file content, returns `null` for missing.
- **Global flag extraction in `.run()`**: Extract flags that need I/O (like `--config <path>`) in `.run()` before calling `.execute()`. Follows same pattern as `env`/`isTTY` auto-sourcing — `.run()` does adapter-dependent work, passes result to pure `.execute()`.
- **CLISchema extension pattern**: Add field with `| undefined` default → update `cli()` factory to include it → add builder method → conditionally act on it in `.run()`/`.execute()`.
- **Config errors before command dispatch**: Detect `--json` from raw argv before config loading to render errors in correct format (JSON vs text).
- **Plugin hook pattern**: Incremental builder method `.configLoader(loader)` appends to `configSettings.loaders`. Requires `.config()` first (throws `INVALID_BUILDER_STATE` otherwise). `configFormat(extensions, parseFn)` convenience factory avoids manual object construction.
- **Metadata modifier pattern**: `.deprecated(message?)` on builders — metadata-only, preserves type-level config. Schema field `deprecated: string | true | undefined`. Warnings collected in resolve (where source is known), returned in `ResolveResult.warnings`, emitted by consumers (`runCommand()`, CLI).

## Task — config-discovery
- New module `src/core/config/index.ts` — pure config file discovery + loading

- Discriminated union result: `ConfigFound | ConfigNotFound`
- 3 error codes: `CONFIG_NOT_FOUND` (explicit path), `CONFIG_PARSE_ERROR`, `CONFIG_UNKNOWN_FORMAT`
- `FormatLoader` interface for custom formats (TOML/YAML); JSON built-in
- `buildConfigSearchPaths()` exported separately for help text / debugging
- `ConfigAdapter = Pick<RuntimeAdapter, 'readFile' | 'cwd' | 'configDir'>` — narrow pick
- Path utilities: `joinPath` (infers separator from base), `getExtension`
- All types + functions exported from `src/index.ts` barrel
- Files: `src/core/config/index.ts` (new), `src/core/config/config.test.ts` (new, 28 tests), `src/index.ts` (barrel)
- **Learnings:** `details` field on `CLIError` is `Record<string, unknown>` — bracket notation required for index access with `noUncheckedIndexedAccess`; biome flags as info but it's correct
