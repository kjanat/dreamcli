## Codebase Patterns

- **Formatter ownership**: dprint owns all formatting (TS/JS via biome plugin, JSON, JSONC, YAML, markdown, etc). Biome's JSON formatter is disabled to avoid conflicts. Biome handles linting only for JSON.
- **JSONC trailing commas**: biome override for JSONC parses trailing commas but does NOT enforce them in formatting (set to "none") to avoid dprint conflicts.
- **Build**: tsdown with `hash: false` for stable dts filenames. `exports: true` auto-patches package.json. Conditional exports include `types` entries for both import and require.
- **Module structure**: `src/core/<module>/index.ts` barrel pattern. `src/runtime/<adapter>.ts` for platform adapters.
- **Test pattern**: vitest with `src/**/*.test.ts` collocated tests. Dynamic imports in tests: `import('./module.js')` (with .js extension for NodeNext).
- **Error hierarchy**: `CLIError` base → `ParseError` (exit 2), `ValidationError` (exit 2). Discriminated by `name` property. `ErrorCode` union is extensible via `string & {}`. Type guards: `isCLIError`, `isParseError`, `isValidationError`.
- **Flag builder pattern**: `FlagBuilder<C extends FlagConfig>` — single phantom type param tracks `valueType` + `presence`. `FlagSchema` is the runtime descriptor. `declare readonly _config: C` for type branding (no `!` assertion). `flag.<kind>()` factory → immutable builder chain. `InferFlag<B>` / `InferFlags<R>` extract resolved types. `const` type param on `enum()` infers literals. Boolean flags implicitly default to `false`.
- **Arg builder pattern**: `ArgBuilder<C extends ArgConfig>` — phantom type tracks `valueType` + `presence` + `variadic`. Positional args default to `'required'` (unlike flags which default to `'optional'`). `arg.custom(parseFn)` infers return type of parse function. Variadic args wrap type in `T[]` via `ResolvedArgValue`. `ArgSchema` stores `parseFn` for custom parsing.
- **Command builder pattern**: `CommandBuilder<F, A>` — two phantom type params accumulate flag/arg builder records via intersection (`F & Record<N, B>`). `.flag()` / `.arg()` return new builders with expanded types; `.action(handler)` receives `ActionParams<F, A>` with `InferFlags<F>` and `InferArgs<A>`. Handler is dropped when new flags/args are added (type safety). `CommandSchema` is the runtime descriptor with flags map + ordered args list. `{}` default for initial accumulator is correct and intentional (biome-ignore). `Out` interface stub defined for handler typing; real impl in output-1.
- **Parser pattern**: Two-phase design: `tokenize(argv)` → `Token[]` (schema-agnostic), then `parse(schema, argv)` → `ParseResult` (schema-aware). Flag lookup built from `CommandSchema.flags` + aliases as `Map<string, [canonicalName, FlagSchema]>`. Value coercion per `FlagKind`/`ArgKind`. Array flags accumulate. Positionals mapped by `CommandSchema.args` ordering. All errors are `ParseError` with structured `code`/`details`/`suggest`.
- **Resolver pattern**: `resolve(schema, parsed)` → `ResolveResult` with `flags` + `args`. MVP chain: CLI → default. Flags resolved first, then args. Missing required values throw `ValidationError`. Multiple errors aggregated into single throw with `details.errors` array. Array flags default to `[]`. Optional flags get `undefined` with key present. Non-empty tuple type `[T, ...T[]]` used for error aggregation to avoid `!` assertions. `isNonEmpty()` type guard narrows `T[]` → `[T, ...T[]]`.
- **Help pattern**: `formatHelp(schema, options?)` → `string`. Reads `CommandSchema` to render sections: Usage, Description, Arguments, Flags, Examples. `HelpOptions` controls `width` (default 80) and `binName`. Flags sorted: short-aliased first, then alphabetical. Type hints per `FlagKind`; boolean suppresses hint and `(default: false)`. Word-wrapping via `wrapText()` at word boundaries with indent continuation.

## Task - parse-1
- Implemented two-phase argv parser: `tokenize()` (schema-agnostic) → `parse()` (schema-aware)
- Tokenizer handles: `--flag`, `--flag=value`, `-f`, `-abc` (combined short), `--` separator, positionals, bare `-` (stdin)
- Parser walks tokens against `CommandSchema`: resolves flag names/aliases via lookup map, coerces values by `FlagKind`/`ArgKind`
- Short flag expansion: `-abc` → individual flags; last value-expecting flag consumes next token; mid-group value-expecting flag consumes rest of chars (`-oFile`)
- Array flags accumulate across multiple occurrences (`--tag v1 --tag v2` → `['v1', 'v2']`)
- Positional args mapped to named entries by schema ordering; variadic args consume all remaining
- "Did you mean?" suggestions via Levenshtein distance for unknown flags
- All errors are `ParseError` with appropriate codes: `UNKNOWN_FLAG`, `MISSING_VALUE`, `INVALID_VALUE`, `UNEXPECTED_POSITIONAL`
- 59 tests covering: tokenizer, all flag kinds, short flags, positional args, variadic, mixed usage, error cases, edge cases
- Re-exported `parse`, `tokenize`, `ParseResult`, `Token` from `src/index.ts`
- Files changed: `src/core/parse/index.ts` (new), `src/core/parse/parse.test.ts` (new), `src/index.ts`
- **Learnings:** TS strict noUncheckedIndexedAccess means `array[i]` returns `T | undefined` — use `charAt()` for strings, `?? fallback` for numeric arrays, or explicit `undefined` guards instead of `as T` assertions. Splitting tokenizer from parser avoids circular concerns: tokenizer is purely lexical, parser is schema-dependent. Biome suppressions that don't match active rules are flagged as `suppressions/unused` warnings.
