# Progress Log

PRD: dreamcli-re-foundation
Started: 2026-04-02

## Codebase Patterns

- The real architectural pressure points are `src/core/cli/`, `src/core/testkit/`, `src/core/resolve/`, `src/core/output/`, and `src/core/completion/`.
- Current execution ownership is effectively centered in `runCommand()` even though the architecture does not present it that way.
- The safest refactor path is contract-first: truth matrix -> characterization tests -> shared executor -> planner split -> resolver redesign.
- Docs work must split into low-risk generated foundations versus semantics-sensitive authored docs.
- `ROADMAP-DOCS-SICK.md` is the main north star for the generated docs platform: examples first, hover on examples only, changelog and docs health, then API inventory, then TypeDoc-based symbol pages.
- Completion breadth must be backed by real implementation and tests; fish and PowerShell are intentional backlog items, not aspirational claims.
- The GitHub project should mirror `prd.json` at mixed depth, with draft items rather than real repo issues.
- When a public type surface is broader than the real support surface, add an audited support-matrix page and make docs state the runtime behavior explicitly instead of staying silent.
- Before comitting, always run `dprint fmt`
- For extraction-first refactors, land a narrow internal contract module plus one reused helper before moving control flow; it gives tests and later module splits a concrete seam without forcing a full rewrite.
- For contract-definition tasks in hot modules, extract the contract to a sibling `contracts.ts` file and have the implementation consume those types; it makes the seam testable without moving behavior yet.
- For policy-heavy modules, extract pure policy helpers that return semantic mode/stream/cleanup facts, then make the renderer consume them; it separates contract from mechanics without forcing a big structural split.
- For planner characterization work, extract one pure `planInvocation()`-style helper that owns root interception and command-plan assembly, then test that seam directly instead of asserting through handlers.
- When subsystem helpers already aggregate their own `ValidationError`s, do cross-domain aggregation at the top boundary by flattening nested error JSON back into `ValidationError`s instead of weakening the lower-level helpers.
