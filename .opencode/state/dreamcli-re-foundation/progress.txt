## Codebase Patterns

- For broad legacy coverage areas, add a small public-boundary contract suite for the exact shipped guarantees instead of piling more cases into the largest existing test file.
- The real architectural pressure points are `src/core/cli/`, `src/core/testkit/`, `src/core/resolve/`, `src/core/output/`, and `src/core/completion/`.
- Current execution ownership is effectively centered in `runCommand()` even though the architecture does not present it that way.
- The safest refactor path is contract-first: truth matrix -> characterization tests -> shared executor -> planner split -> resolver redesign.
- When execution ownership is implicit, add a dedicated black-box contract suite at the current owner boundary before extraction; it lets later shared seams move without coupling tests to module shape.
- Docs work must split into low-risk generated foundations versus semantics-sensitive authored docs.
- `ROADMAP-DOCS-SICK.md` is the main north star for the generated docs platform: examples first, hover on examples only, changelog and docs health, then API inventory, then TypeDoc-based symbol pages.
- Completion breadth must be backed by real implementation and tests; fish and PowerShell are intentional backlog items, not aspirational claims.
- The GitHub project should mirror `prd.json` at mixed depth, with draft items rather than real repo issues.
- When a public type surface is broader than the real support surface, add an audited support-matrix page and make docs state the runtime behavior explicitly instead of staying silent.
- Before comitting, always run `dprint fmt`
- For extraction-first refactors, land a narrow internal contract module plus one reused helper before moving control flow; it gives tests and later module splits a concrete seam without forcing a full rewrite.
- For contract-definition tasks in hot modules, extract the contract to a sibling `contracts.ts` file and have the implementation consume those types; it makes the seam testable without moving behavior yet.
- For policy-heavy modules, extract pure policy helpers that return semantic mode/stream/cleanup facts, then make the renderer consume them; it separates contract from mechanics without forcing a big structural split.
- For planner characterization work, extract one pure `planInvocation()`-style helper that owns root interception and command-plan assembly, then test that seam directly instead of asserting through handlers.
- When subsystem helpers already aggregate their own `ValidationError`s, do cross-domain aggregation at the top boundary by flattening nested error JSON back into `ValidationError`s instead of weakening the lower-level helpers.
- For execution refactors, extract a minimal `executeCommand()` seam with explicit request/result types and a shared `buildRunResult()` helper first; keep CLI and testkit as thin bootstrap wrappers until routing work lands.
- For oversized hot modules, keep `index.ts` as the orchestration boundary and extract sibling concern modules first (`args`, `flags`, `coerce`, `config`, `errors`) before changing behavior; the contract tests stay stable while the file shape improves.
- When centralizing execution ownership, wrap the whole executor in one `try/finally` so fast exits like `--help` and missing-handler guards still clean up active output handles.
- When a new internal seam broadens a hot test helper interface, make the seam optional and keep the old path as a fallback for fake test doubles; production wiring can move first without forcing broad fixture churn.
- For output-layer policy refactors, freeze one `policy` snapshot per channel and route concrete writer or handle creation through sibling renderer helpers; that splits semantic mode choice from mechanics without changing the public `Out` surface.
- For CLI run-path refactors, extract adapter-driven config/package/stdin/prompt sourcing into a `runtime-preflight` helper that consumes planner outcomes; it removes duplicate dispatch branching while keeping `run()` as a thin runtime shell.
- Once a planner seam exists, pass raw argv into it and let the planner own global invocation normalization like root `--json`; keep `execute()` focused on output setup, rendering, and executor handoff.
- For generated-docs groundwork, keep source-backed build outputs under `docs/.generated/` and let authored wrapper pages plus VitePress config import one generated manifest; that preserves clean source boundaries while future docs generators can expand without reshaping IA again.
- For generated reference work, build one source-backed inventory model first and emit both JSON and markdown artifacts from it; authored wrapper pages can stay on stable routes while later normalization work reuses the same data seam.
- For TypeDoc-backed reference work, keep the public export inventory as coverage truth, generate raw `typedoc.json` beside a smaller DreamCLI-owned normalized model keyed by `entrypoint:symbol`, and force stable source links so generated artifacts do not churn on every commit.
- When flags and args partially overlap, share only the coercion-compatible property shape in a dedicated resolver helper and keep precedence, fallback, and required-value rules separate; broad shared property loops hide real semantic differences.
- For generated reference work, keep the heavy source-of-truth model under `docs/.generated/`, but emit lightweight searchable public routes from that model and keep VitePress metadata limited to route/index facts instead of full page payloads.
- For generated example docs, render full source-backed detail pages to `docs/examples/*.md`, but keep `docs/.generated/site-data.ts` limited to nav/index metadata; pushing full source blobs into VitePress config metadata creates avoidable churn.
- For resolver diagnostic redesign, keep nested `ValidationError` JSON stable and add a parallel per-issue summary at the aggregate wrapper; it clarifies mixed flag/arg failures without weakening existing leaf-error assertions.
- For generated docs routes, detect case-insensitive slug collisions at generation time and suffix only the colliding non-lowercase export; VitePress lowercases page keys during render, so case-only route differences are unstable.
- For docs interactivity rollouts, keep generated example pages readable without hover by putting `ts twoslash` only on the source block and leaving summary, usage, and related-link sections plain.
- For generated docs surfaces that need public routes, keep the source-of-truth artifact under `docs/.generated/` and emit the browsable `docs/reference/*.md` page from the same shared renderer so metrics and public copy stay in lockstep.
- For source-backed schema metadata, derive one generated TS description map from the normalized TypeDoc model and let runtime exports consume that generated module; it keeps the shipped schema asset and in-process constant aligned without hand-written doc strings.
