# v0.7 — Subcommand Nesting (Command Trees)

## Problem

DreamCLI currently supports only flat, single-level commands: `cli > command`. Real-world CLIs need
arbitrary depth: `mycli deploy staging --force`, `kubectl get pods -n default`,
`docker compose up -d`. Without nesting, users must flatten everything into top-level commands with
awkward naming (`deploy-staging`, `deploy-production`) or use positional args as pseudo-subcommands.

## Design Decisions

### D1: Unlimited recursive depth

No artificial depth cap. The type system erases at command boundaries (each command is
`ErasedCommand` internally), so there's no TS recursion risk regardless of depth. Each command's
action handler is independently typed — the parent doesn't need to know the child's flag types.

### D2: Optional action on groups

A command with children can optionally have its own action handler. If invoked without a recognized
subcommand:

- **Has action**: run it (remaining argv becomes that command's positional args)
- **No action**: show help for the group (list subcommands)

This matches `docker compose` (has default behavior) vs `kubectl config` (shows help).

### D3: Explicit opt-in flag propagation via `.propagate()`

Flags don't inherit by default. A flag marked `.propagate()` on a parent command becomes available
to all descendants. Child can override by redeclaring a flag with the same name — child definition
wins completely (no merging).

**Rationale**: The flag definition is the single source of truth. Whether it propagates is a
property *of the flag*, not of the parent-child relationship. This avoids coordination problems
where renaming a flag requires updating separate propagation lists.

```ts
const verbose = flag.boolean().alias('v').description('Verbose output').propagate();

const app = cli('myapp')
	.flag('verbose', verbose)
	.command(deploy) // gets --verbose automatically
	.command(status); // gets --verbose automatically
```

Override by redeclaring:

```ts
const canary = command('canary').flag('verbose', flag.enum(['minimal', 'full']).default('minimal')); // overrides parent's boolean
```

### D4: Type erasure at depth boundaries

Children are `ErasedCommand` internally. Type inference is per-command only. The parent knows a
child *exists* (name, description for help) but not its internal flag/arg types. This is the tRPC
pattern: each procedure typed independently, router just collects them.

```ts
const staging = command('staging')
	.flag('replicas', flag.number().default(1))
	.action(({ flags }) => {
		flags.replicas; // number
		flags.verbose; // boolean (propagated from ancestor)
	});

const deploy = command('deploy').flag('verbose', flag.boolean().propagate()).command(staging); // ErasedCommand — parent just knows it exists
```

### D5: Dual API — `group()` factory + `.command()` inline

Both patterns supported:

```ts
// Inline nesting
const deploy = command('deploy')
	.description('Deployment commands')
	.command(staging)
	.command(production);

// group() convenience (equivalent, just clearer intent)
const deploy = group('deploy', [staging, production]).description('Deployment commands');
```

`group()` is sugar — creates a `CommandBuilder` with children pre-registered and no action.

### D6: Group help shows flags + subcommands

`mycli deploy --help` shows both the group's own flags (if any) and its subcommands:

```
mycli deploy --help

Deploy to an environment

Usage: mycli deploy [flags] <command>

Flags:
  -v, --verbose    Verbose output
  -f, --force      Skip confirmation

Commands:
  staging          Deploy to staging
  production       Deploy to production
  canary           Deploy canary release

Run "mycli deploy <command> --help" for more info
```

## Schema Changes

### FlagSchema

```diff
  interface FlagSchema {
    // ... existing fields ...
    readonly deprecated: string | true | undefined;
+   /** Whether this flag propagates to child commands. */
+   readonly propagate: boolean;
  }
```

### CommandSchema

```diff
  interface CommandSchema {
    // ... existing fields ...
    readonly middleware: readonly ErasedMiddlewareHandler[];
+   /** Child commands (empty array for leaf commands). */
+   readonly commands: readonly CommandSchema[];
  }
```

### ErasedCommand

No changes needed — `schema.commands` carries the children.

## Dispatch Algorithm

Recursive walk consuming argv segments:

```
function resolveCommandPath(commands, argv):
  path = []
  remaining = argv
  current = commands  // top-level command list

  while remaining.length > 0:
    candidate = remaining[0]
    match = findCommand(current, candidate)

    if match:
      path.push(match)
      remaining = remaining.slice(1)
      current = match.schema.commands
    else:
      break  // candidate is not a command name — it's a flag or arg

  // path = [deploy, staging], remaining = ['--force', 'target']
  leaf = path[path.length - 1] ?? null

  if leaf is null:
    // No command matched — show root help or error
  else if leaf.schema.commands.length > 0 && !leaf.schema.hasAction:
    // Group without action — show group help
  else:
    // Leaf (or group with action) — collect propagated flags from ancestors, execute
    propagatedFlags = collectPropagatedFlags(path)
    leaf._execute(remaining, { propagatedFlags })
```

### Propagated flag collection

Walk the command path from root to parent (exclusive of leaf). For each ancestor, collect flags
where `propagate === true`. If a descendant redeclares the same name, the descendant's definition
wins (shadowing).

```
function collectPropagatedFlags(path):
  accumulated = {}
  // path[0..n-2] are ancestors, path[n-1] is the target command
  for ancestor in path.slice(0, -1):
    for [name, flagSchema] of ancestor.schema.flags:
      if flagSchema.propagate:
        accumulated[name] = flagSchema  // later ancestors shadow earlier
  // Remove any flag the leaf itself declares (leaf's own definition wins)
  for name of leaf.schema.flags:
    delete accumulated[name]
  return accumulated
```

## Completion Changes

Both bash and zsh generators must recursively walk the command tree. The current single-level
`case "$subcmd"` pattern becomes a loop tracking command depth:

- **Bash**: `for ((i=1; i < COMP_CWORD; i++))` loop walking `${COMP_WORDS[i]}` to find the deepest
  matched command, then complete from that scope.
- **Zsh**: Nested `_arguments -C` with `->state` dispatch per command level.

## Risks

1. **cli/index.ts already ~700 lines** — recursive dispatch adds significant complexity. May need to
   extract dispatch logic into a separate module (e.g. `src/core/cli/dispatch.ts`). Mitigate by
   extracting `resolveCommandPath()` and `collectPropagatedFlags()` as pure functions.

2. **Completion generator complexity** — Recursive tree walking in bash/zsh generation is
   error-prone. Mitigate with comprehensive test coverage using multi-level command trees.

3. **Propagated flag resolution ordering** — When a propagated flag has `.env()` or `.config()`,
   resolution still follows the normal chain (CLI > env > config > prompt > default). The flag
   schema is just merged into the child's flag set before resolution runs. No special resolution
   logic needed.

## Non-goals

- **Cross-command flag sharing without propagation**: If two sibling commands need the same flag,
  they each declare it. No "shared flag" concept beyond parent propagation.
- **Nested middleware propagation**: Middleware doesn't propagate to children. Each command declares
  its own middleware chain. (Future consideration for v1.0 plugin hooks.)
- **Nested config namespacing**: Config paths are absolute dotted paths, not relative to command
  tree position. `flag.config('deploy.staging.replicas')` not `flag.config('.replicas')`.
