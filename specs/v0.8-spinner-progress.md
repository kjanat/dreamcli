# v0.8 — Spinner & Progress on Out

## Problem

DreamCLI's `Out` interface covers text, JSON, and table output but has no support for transient
terminal feedback — spinners for indeterminate waits, progress bars for measured work. Handlers that
need these today must reach for external packages, manage TTY detection themselves, and remember to
suppress decorations in `--json` mode. This is exactly the kind of cross-cutting concern the output
channel should own.

## Design Decisions

### D1: Handle-based spinner with `.wrap()` convenience

Primary API returns a `SpinnerHandle` with explicit lifecycle control. A `.wrap(promise)` shorthand
auto-starts and auto-stops (succeed on resolve, fail on reject). The handle is the primitive;
`.wrap()` is sugar that prevents "forgot to stop" bugs.

```ts
// Handle-based — full control
const s = out.spinner('Deploying...');
s.update('Uploading artifacts...');
s.update('Running migrations...');
s.succeed('Deployed in 4.2s');

// Wrap-a-promise — 80% use case
await out.spinner('Building...').wrap(build());

// Wrap with custom messages
await out.spinner('Installing deps...').wrap(install(), {
	succeed: 'Installed 142 packages',
	fail: 'Installation failed',
});
```

Handle methods: `.update(text)`, `.succeed(text?)`, `.fail(text?)`, `.stop()`. Calling any terminal
method (succeed/fail/stop) after the handle is already stopped is a no-op — not an error.

### D2: Determinate + indeterminate progress

Progress bar with optional `total`. When `total` is provided: determinate mode (shows percentage,
filled bar). When omitted: indeterminate mode (pulsing/bouncing indicator). Indeterminate is trivial
to implement — it's "spinner but shaped like a bar."

```ts
// Determinate
const p = out.progress({ total: files.length, label: 'Processing' });
for (const file of files) {
	await process(file);
	p.increment(); // +1
}
p.done();

// Determinate with explicit value
const p = out.progress({ total: 100, label: 'Upload' });
p.update(42); // jump to 42/100
p.done('Upload complete');

// Indeterminate (no total)
const p = out.progress({ label: 'Scanning...' });
// ... work ...
p.done();
```

Handle methods: `.increment(n?)`, `.update(value)`, `.done(text?)`, `.fail(text?)`. Same
terminal-method-is-idempotent rule as spinner.

No multi-bar support — that's cursor management complexity for v1.0+.

### D3: Configurable non-TTY fallback (default: silent)

When `!isTTY` or `jsonMode`, spinner/progress calls are no-ops by default. No output, no side
effects. This is correct for CI where noise is harmful.

Per-call opt-in for static fallback via `{ fallback: 'static' }`:

```ts
// Silent in CI (default)
const s = out.spinner('Deploying...');

// Static log lines in CI
const s = out.spinner('Deploying...', { fallback: 'static' });
// non-TTY output:
//   ⏳ Deploying...
//   ✓ Deployed successfully
```

Static fallback emits plain text via `out.log()` / `out.error()` at start and terminal events only
(not on every `.update()`). This means static fallback output appears in captured stdout/stderr
arrays like any other log line.

### D4: Typed activity events in testkit

Testkit captures spinner/progress lifecycle as a dedicated `result.activity` array of discriminated
union events:

```ts
type ActivityEvent =
	| { readonly type: 'spinner:start'; readonly text: string }
	| { readonly type: 'spinner:update'; readonly text: string }
	| { readonly type: 'spinner:succeed'; readonly text: string }
	| { readonly type: 'spinner:fail'; readonly text: string }
	| { readonly type: 'spinner:stop' }
	| { readonly type: 'progress:start'; readonly label: string; readonly total: number | undefined }
	| { readonly type: 'progress:update'; readonly value: number }
	| { readonly type: 'progress:done'; readonly text: string | undefined }
	| { readonly type: 'progress:fail'; readonly text: string | undefined };
```

This keeps activity separate from stdout/stderr (no polluted assertions), preserves ordering within
the activity stream, and composes well — filter by type, check ordering, ignore what you don't care
about.

### D5: TTY rendering strategy

When `isTTY && !jsonMode`, spinners and progress bars render with ANSI cursor control:

- **Spinner**: Writes current frame + text on one line, overwrites with `\r` on each tick. On
  terminal event, clears line and writes final message with status symbol (✓/✗).
- **Progress**: Writes `[████░░░░] 42% label` on one line, overwrites with `\r`. On done, clears and
  writes completion message.

Frame animation uses a configurable interval (default 80ms). The timer is driven by `setInterval` —
no external deps needed. The handle's `.stop()`/`.succeed()`/`.fail()` clears the interval.

**No concurrent spinner/progress overlap.** Starting a new spinner while one is active implicitly
stops the previous one. Starting progress while a spinner is active stops the spinner. This avoids
cursor collision without multi-line cursor management.

### D6: Zero runtime dependencies

Spinner frames (braille dots / line chars), progress bar characters, and ANSI escape sequences are
all inline string literals. No dependency on `ora`, `cli-spinners`, `cli-progress`, or similar.

## Scope

### In scope

- `out.spinner(text, options?)` → `SpinnerHandle`
- `out.progress(options)` → `ProgressHandle`
- TTY rendering with ANSI cursor control
- Silent no-op for non-TTY/jsonMode (default)
- Static fallback opt-in per call
- `result.activity` in `RunResult` with typed `ActivityEvent[]`
- `CapturedOutput.activity` extension
- Tests for all modes: TTY, non-TTY, jsonMode, static fallback, testkit capture

### Out of scope (explicit non-goals)

- Multi-bar / concurrent progress bars
- Color/style helpers (`out.bold()`, `out.dim()`)
- Persistent status bar (`out.status()`)
- Custom spinner frame sets (use the one built-in set)
- Streaming JSON events for activity in `--json` mode

## Type Additions

### On `Out` interface (`schema/command.ts`)

```ts
interface Out {
	// ... existing methods ...

	/**
	 * Create a spinner for indeterminate progress feedback.
	 *
	 * Returns a handle for lifecycle control. In non-TTY/jsonMode,
	 * returns a no-op handle (or static fallback if configured).
	 */
	spinner(text: string, options?: SpinnerOptions): SpinnerHandle;

	/**
	 * Create a progress bar for measured work.
	 *
	 * Returns a handle for updating progress. Pass `total` for
	 * determinate mode; omit for indeterminate (pulsing).
	 */
	progress(options: ProgressOptions): ProgressHandle;
}
```

### New types (in `schema/command.ts` or new `schema/activity.ts`)

```ts
type Fallback = 'silent' | 'static';

interface SpinnerOptions {
	readonly fallback?: Fallback;
}

interface SpinnerHandle {
	update(text: string): void;
	succeed(text?: string): void;
	fail(text?: string): void;
	stop(): void;
	wrap<T>(
		promise: Promise<T>,
		options?: {
			readonly succeed?: string;
			readonly fail?: string;
		},
	): Promise<T>;
}

interface ProgressOptions {
	readonly total?: number;
	readonly label?: string;
	readonly fallback?: Fallback;
}

interface ProgressHandle {
	increment(n?: number): void;
	update(value: number): void;
	done(text?: string): void;
	fail(text?: string): void;
}
```

### On `RunResult` (testkit)

```ts
interface RunResult {
	// ... existing fields ...
	readonly activity: readonly ActivityEvent[];
}
```

| `src/index.ts`                | Export new public types                                                     | S      |
| Tests (new files)             | `output-spinner.test.ts`, `output-progress.test.ts`, `activity-e2e.test.ts` | M      |

### Implementation layers

```
Out interface (schema)     → contract: spinner(), progress()
    ↓
OutputChannel (output)     → dispatch to appropriate handle factory
    ↓
┌─────────────────────────────────────────────────┐
│ isTTY && !jsonMode → TTYSpinnerHandle           │
│                    → TTYProgressHandle          │
│ !isTTY || jsonMode →                            │
│   fallback=static  → StaticSpinnerHandle        │
│                    → StaticProgressHandle       │
│   fallback=silent  → NoopSpinnerHandle          │
│                    → NoopProgressHandle         │
└─────────────────────────────────────────────────┘
    ↓
CaptureSpinnerHandle / CaptureProgressHandle (testkit mode)
    → records ActivityEvent[] into CapturedOutput.activity
    → delegates to underlying handle (noop in capture mode)
```

Capture handles wrap whichever handle the mode selects, intercepting every method call to record the
event before delegating. This means testkit always captures events regardless of TTY/jsonMode
settings.

### Implicit stop on overlap (D5)

`OutputChannel` tracks the "current active handle" (spinner or progress). When `.spinner()` or
`.progress()` is called while one is active, the active handle is `.stop()`'d first. This is a
simple state machine:

```
idle ──spinner()──→ spinner_active ──spinner()──→ (stop prev) spinner_active
  │                      │                              ↑
  │                  succeed/fail/stop                  │
  │                      ↓                              │
  └──progress()──→ progress_active ──spinner()──→ (stop prev) spinner_active
                       │
                   done/fail
                       ↓
                     idle
```

## Deliverables (Ordered)

1. **[D1] Activity types** (S) — `ActivityEvent`, `SpinnerHandle`, `ProgressHandle`,
   `SpinnerOptions`, `ProgressOptions`, `Fallback` types. Add `spinner()` and `progress()` to `Out`.
   Depends on: nothing.

2. **[D2] Noop + static handles** (S) — `NoopSpinnerHandle`, `NoopProgressHandle`,
   `StaticSpinnerHandle`, `StaticProgressHandle`. These are the non-TTY implementations. Depends on:
   D1.

3. **[D3] Capture handles + testkit wiring** (M) — `CaptureSpinnerHandle`, `CaptureProgressHandle`
   that record `ActivityEvent[]`. Extend `CapturedOutput` and `RunResult` with `activity`. Wire
   through `createCaptureOutput()` and `runCommand()`. Depends on: D2.

4. **[D4] TTY handles + ANSI rendering** (L) — `TTYSpinnerHandle`, `TTYProgressHandle`. Spinner
   frame animation, progress bar rendering, `\r` overwrite, cursor show/hide. Timer management.
   Depends on: D1.

5. **[D5] OutputChannel integration** (M) — Implement `spinner()` and `progress()` on
   `OutputChannel`. Mode dispatch (TTY vs noop vs static). Active handle tracking + implicit stop.
   Depends on: D2, D3, D4.

6. **[D6] Public API exports** (S) — Export new types from `schema/index.ts` and `src/index.ts`.
   Depends on: D1.

7. **[D7] Tests** (M) — Unit tests for each handle type. Integration tests via testkit. Coverage:
   TTY mode, non-TTY silent, non-TTY static fallback, jsonMode suppression, implicit stop on
   overlap, `.wrap()` success/failure, activity event capture, progress determinate/indeterminate.
   Depends on: D5, D6.

## Risks

1. **Timer cleanup on unhandled rejection** — If a handler throws while a spinner is active, the
   `setInterval` leaks. Mitigation: `runCommand()` wraps handler execution and calls `.stop()` on
   the active handle in a `finally` block.

2. **Test flakiness from timing** — TTY spinner uses `setInterval(80ms)`. Tests must not depend on
   real timers. Mitigation: TTY handle tests use `vi.useFakeTimers()` to control tick; capture
   handle tests are timer-free (events recorded synchronously).
